# QuantImage v2 – Frontend Performance Monitoring Suite

A collection of lightweight monitoring scripts to diagnose frontend performance issues during multi-user sessions.

## Quick Start

```bash
cd monitoring/

# Start all monitors with default intervals
./start-monitoring.sh

# Check status
./start-monitoring.sh --status

# Stop all monitors
./start-monitoring.sh --stop

# Analyze collected data after the session
./11-analyze-logs.sh
```

## Profiles

| Profile | Flag | Impact | Use Case |
|---------|------|--------|----------|
| **Light** | `--light` | ~1-2% CPU | Long sessions, resource-constrained servers |
| **Normal** | *(default)* | ~3-5% CPU | Typical monitoring sessions |
| **Heavy** | `--heavy` | ~5-8% CPU | Short debugging sessions, more granular data |

## Scripts Overview

### Continuous Monitors (run in background)

| # | Script | What it monitors | Default Interval |
|---|--------|-----------------|-----------------|
| 01 | `01-host-resources.sh` | CPU, memory, swap, disk I/O, network, load avg | 10s |
| 02 | `02-docker-containers.sh` | CPU/mem/network per container (quantimage2, kheops, keycloak, traefik) | 15s |
| 03 | `03-api-latency.sh` | DNS, connect, TTFB, total time for frontend/backend/kheops/keycloak | 15s |
| 04 | `04-nginx-connections.sh` | Nginx active connections, worker count, request rate | 15s |
| 05 | `05-socketio-health.sh` | Socket.IO polling/WS transport, Redis pub/sub health | 20s |
| 08 | `08-traefik-access.sh` | Request rate, response time percentiles, error rates | 30s |
| 09 | `09-self-monitor.sh` | CPU/memory of monitoring scripts themselves | 30s |
| 10 | `10-backend-health.sh` | Flask, Celery workers, Redis queues, MySQL connections | 20s |

### One-Shot Analysis

| # | Script | Purpose |
|---|--------|---------|
| 07 | `07-bundle-analysis.sh` | JS/CSS bundle sizes, gzip ratios, nginx config audit, cache headers |
| 11 | `11-analyze-logs.sh` | Aggregate analysis of all collected CSV logs into a report |

### Browser-Side Monitor

| # | Script | Purpose |
|---|--------|---------|
| 06 | `06-browser-performance.js` | Paste into DevTools console – tracks fetch timing, long tasks, CLS, memory, DOM complexity, Socket.IO events |

## Typical Workflow for Thursday's Session

### Before users arrive

```bash
# 1. Run bundle analysis to check for obvious issues
./start-monitoring.sh --bundle

# 2. Start all monitors in light mode
./start-monitoring.sh --light
```

### During the session

```bash
# Check status and resource overhead
./start-monitoring.sh --status

# (Optional) Open browser DevTools on your machine and paste 06-browser-performance.js
# Then interact with the app normally
```

### After the session

```bash
# 1. Stop monitors
./start-monitoring.sh --stop

# 2. Generate analysis report
./11-analyze-logs.sh

# 3. Check browser data (if you used the browser monitor)
# In DevTools console:
#   __QI2_PERF_MONITOR__.getSummary()
#   __QI2_PERF_MONITOR__.exportJSON()
```

## What Each Monitor Detects

### Frontend-Specific Issues
- **Missing gzip compression** → bundle analysis checks nginx config
- **No cache headers** → bundle analysis checks live response headers
- **Source maps in production** → bundle analysis flags `.map` files
- **Large bundles** → bundle analysis lists sizes with gzip estimates
- **Slow page loads** → API latency tracks TTFB for frontend static files
- **WebSocket issues** → Socket.IO health checks transport availability
- **Main thread jank** → browser monitor tracks long tasks (>50ms)
- **Memory leaks** → browser monitor samples JS heap periodically
- **Layout thrashing** → browser monitor tracks Cumulative Layout Shift
- **Slow API calls** → browser monitor intercepts all `fetch()` calls with timing

### Infrastructure Issues
- **CPU saturation** → host resources and container-level CPU tracking
- **Memory pressure** → host and per-container memory trends
- **Disk I/O bottleneck** → host I/O wait percentage
- **Network congestion** → host network throughput
- **Container resource contention** → per-container CPU/memory over time

### Backend Issues (affecting frontend)
- **Slow API responses** → API latency probe + browser fetch interceptor
- **Celery queue backlog** → backend health checks Redis queue depth
- **Database bottleneck** → MySQL connection count monitoring
- **Socket.IO server issues** → dedicated Socket.IO health check

## Customization

### Running individual scripts

Each script can be run standalone with a custom interval:

```bash
# Monitor only API latency every 5 seconds
./03-api-latency.sh 5

# Monitor only Docker containers every 30 seconds
./02-docker-containers.sh 30
```

### Adjusting thresholds

Alert thresholds are hardcoded in each script. Key ones:
- Host CPU > 80% → alert in `01-host-resources.sh`
- API response > 2s → yellow in `03-api-latency.sh`
- Backend CPU > 80% → alert in `10-backend-health.sh`
- Celery queue > 10 → alert in `10-backend-health.sh`
- Monitoring overhead > 10% CPU → alert in `09-self-monitor.sh`

## Resource Usage of Monitoring Scripts

The monitoring suite is designed to be lightweight:

| Component | CPU Impact | Memory Impact | Disk I/O |
|-----------|-----------|--------------|----------|
| Host monitor | ~0.2%/sample | <5 MB | ~1 KB/sample |
| Docker monitor | ~1-2%/sample | <10 MB | ~2 KB/sample |
| API latency | ~0.1%/sample | <5 MB | ~0.5 KB/sample |
| Nginx monitor | ~0.5%/sample | <5 MB | ~0.2 KB/sample |
| Socket.IO health | ~0.3%/sample | <5 MB | ~0.3 KB/sample |
| Traefik access | ~0.5%/sample | <5 MB | ~0.5 KB/sample |
| Self-monitor | ~0.1%/sample | <5 MB | ~0.5 KB/sample |
| Backend health | ~1%/sample | <5 MB | ~0.5 KB/sample |
| **Total (light)** | **~2-3%** | **~40 MB** | **~50 KB/min** |
| **Total (normal)** | **~3-5%** | **~40 MB** | **~100 KB/min** |

The `09-self-monitor.sh` script tracks actual resource usage so you can verify these estimates.
